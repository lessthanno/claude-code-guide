同样的需求，不同的提问方式，代码质量差距可能有 10 倍。这不是玄学，而是有规律可循的。

**模式 1：约束驱动（先说不能做什么）**

差的提法：
```
帮我写一个缓存函数
```

好的提法：
```bash
claude "实现一个 Go 泛型缓存函数，约束：
- 不使用第三方库
- 线程安全（支持并发读写）
- TTL 过期自动清理（不是查询时才清理，而是有后台 goroutine）
- 内存占用可控（最大条目数限制，LRU 淘汰）
- 不接受 interface{}，用泛型 K comparable, V any"
```

明确说不能做什么，比只说要做什么更能引导出高质量代码。

**模式 2：提供上下文（为什么要写这个）**

差的提法：
```
写一个 retry 函数
```

好的提法：
```bash
claude "我需要一个 retry 函数，用于我们的 HTTP 客户端。

背景：
- 调用第三方 AI API，偶尔会 503
- 我们希望对 5xx 错误重试，对 4xx 不重试
- 已有 context 传递（支持超时取消）
- 使用指数退避 + jitter（防止雷群效应）

参考：我们其他重试代码用 time.Sleep，没有 jitter
这次希望改进这一点。"
```

**模式 3：给出反例（不要什么样的）**

```bash
claude "实现一个 HTTP 中间件，记录请求日志。

不要这样做：
- 不要用 fmt.Println（用 slog）
- 不要记录请求 body（性能问题 + PII 风险）
- 不要用全局变量存 logger

要这样做：
- logger 通过 context 传递
- 记录：method、path、status、latency、request_id
- 敏感路径（/health、/metrics）不记录
- 结构化 JSON 日志"
```

**模式 4：分层实现（先框架后细节）**

```bash
# 第一步：只要接口
claude "定义这个支付服务的 Go 接口：
PaymentService，需要支持：
创建支付、查询支付、取消支付、退款
只给接口定义和注释，不要实现。"

# 第二步：实现一个方法
claude "实现上面接口的 CreatePayment 方法。
接口定义：[上面的输出]
实现要求：
- 调用 Stripe SDK
- 幂等性处理（idempotency_key）
- 错误类型映射（Stripe 错误 → 业务错误）"

# 第三步：测试
claude "为 CreatePayment 写单元测试。
实现代码：[上面的输出]
测试覆盖：成功、Stripe 错误、网络超时、重复请求"
```

**模式 5：现有代码风格匹配**

```bash
claude "在这个项目里添加一个新的 API handler。

现有 handler 示例（学习这种风格）：
[粘贴 2-3 个现有 handler]

新 handler 的功能：
GET /api/users/:id/stats
返回：用户的文章数、评论数、最后活跃时间

严格按照现有代码的风格：
- 错误处理方式
- 响应格式
- 中间件的使用
- 日志的写法"
```

**模式 6：测试先行**

```bash
claude "先帮我写这个函数的测试用例，不要实现：

函数功能：解析 Cron 表达式，返回下一次执行时间
输入：cron string + 当前时间
输出：下一次时间 + error

测试覆盖：
- 标准格式（* * * * *）
- 带具体值（0 9 * * 1-5，工作日 9 点）
- 边界情况（2 月 29 日、年末、无效表达式）
- 已知的 corner case"
```

最后，所有这些模式的本质是同一件事：**让 Claude 理解你的真实约束，而不是你想要结果的形状**。你的约束（线程安全、不用第三方库、匹配现有风格）包含了大量关于你的系统的隐式知识——把这些约束说出来，代码质量会大幅提升。
