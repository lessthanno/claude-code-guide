正则表达式是"写得了读不懂"的典型例子。Claude Code 在这里最直接的价值是：**从你描述的意图生成正则，并解释现有正则的含义**。

**从意图生成正则：**

```bash
claude "帮我写一个正则表达式：

匹配：有效的中国大陆手机号
规则：
- 1 开头
- 第二位是 3-9（涵盖所有运营商）
- 总共 11 位数字
- 可能有 +86 前缀或 86 前缀
- 中间可以有 - 或空格分隔（如 138-1234-5678）

给出：
1. 正则表达式（Go 语法）
2. 解释每个部分的含义
3. 测试用例（合法的和不合法的）
4. 常见的 edge case 这个正则会不会误判？"
```

**解读复杂正则：**

```bash
claude "解释这个正则表达式在做什么：
^(?:(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\.){3}(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)$

1. 逐部分解释（不要跳过任何括号）
2. 它匹配什么，不匹配什么？
3. 有什么已知的局限性？
4. 用更可读的方式改写（带注释的多行版本）"
```

**日志解析：**

```bash
claude "帮我解析这种格式的 Nginx access log：

格式：
192.168.1.1 - - [01/Jan/2026:00:00:00 +0800] \"GET /api/users HTTP/1.1\" 200 1234 \"https://example.com\" \"Mozilla/5.0...\"

要提取：
- IP、时间、HTTP 方法、路径、状态码、响应体大小
- 用 Go 实现
- 要处理：referer 和 user-agent 可能包含引号和转义字符

给出 Go 结构体 + 解析函数 + 测试用例。"
```

**数据清洗：**

```bash
claude "帮我写这个数据清洗脚本：

输入：用户填写的地址，格式混乱
例子：
- '北京市朝阳区三里屯路10号'
- '北京朝阳三里屯10号'
- '朝阳区三里屯10号 北京'
- 'Beijing, Chaoyang, Sanlitun No.10'

目标：
1. 提取省/市/区/街道/门牌号
2. 标准化格式（去掉重复、统一顺序）
3. 置信度评分（能确定的 vs 猜测的）

这是 NLP 问题，不是纯正则问题。
帮我设计方案：纯规则 vs 用 Claude API 做提取？
什么情况下哪种更合适？"
```

**配置文件解析（非标准格式）：**

```bash
claude "帮我解析这个非标准格式的配置文件：

[示例文件内容：
database.host = localhost
database.port=5432
# 注释行
cache {
  ttl = 300
  max_size = 1000
}
]

用 Go 解析成 map[string]interface{}：
1. 支持嵌套（用 . 分隔的 key）
2. 忽略注释（# 开头）
3. 支持引号字符串（值可以是 \"hello world\"）
4. 类型推断（数字不用引号）

给出解析器实现 + 完整测试。"
```

**URL 操作：**

```bash
claude "帮我实现这些 URL 操作函数（Go）：

1. 提取 domain：
   'https://www.example.com/path?q=1' → 'example.com'
   
2. 添加 query 参数（保留现有的）：
   addParam('https://example.com?a=1', 'b', '2') → 'https://example.com?a=1&b=2'

3. 标准化 URL（小写 domain、移除默认端口、排序 query 参数）

4. 判断 URL 是否同源

不要用第三方库，用标准库 net/url 实现。
每个函数给出实现 + 边界情况测试。"
```

文本处理的 Claude Code 原则：**描述意图，不要描述正则语法**。"匹配 IP 地址"比"匹配四组 1-3 位数字用点分隔"更能让 Claude 给出正确且考虑边界情况的答案。正则是实现细节，意图才是要描述的东西。
