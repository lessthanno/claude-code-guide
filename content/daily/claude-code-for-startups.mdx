3 个工程师做 10 个工程师的活儿，是很多早期创业公司的现实。Claude Code 在这个场景下不是"提升效率的工具"，而是"能力的乘数"。

**早期公司最适合用 Claude Code 的场景：**

**1. 快速验证技术可行性**

```bash
claude "我们要实现实时位置共享，
用 WebSocket 还是 SSE 更适合这个场景（移动端为主）？
列出关键差异，给出推荐方案，
包括大概的实现复杂度评估。"
```

3 分钟而不是 3 小时做技术选型。

**2. 搭建基础设施**

早期公司经常在基础设施上花太多时间。Claude Code 帮你快速搭出"够用"的版本：

```bash
claude "我们需要一个后台任务处理系统。
需求：
- 每天处理约 10000 个任务
- 任务失败需要重试（最多 3 次）
- 需要监控哪些任务失败了
- 团队只有 Go 技术栈，不想引入新语言

不要过度设计，给我能在 1 天内实现的方案。"
```

**3. 一个人做多个角色**

小团队里，工程师经常需要跨领域工作：

```bash
# 工程师写文案
claude "为这个功能写一段 App Store 描述（100 字），
功能：用户可以实时分享位置给好友，
并设置共享时限。"

# 工程师设计数据库
claude "我们的产品是一个...
帮我设计初始的数据库 schema，
只包含 MVP 需要的表，
不要过度设计。"
```

**4. 快速 onboard 新成员**

```bash
claude "新工程师明天加入。
基于我们的代码库，
生成一份'第一周任务清单'：
- 第 1-2 天：了解代码库的任务
- 第 3-4 天：在现有功能上做小改动的任务
- 第 5 天：独立完成一个小功能

任务要具体，包含要读的文件和要做的修改。"
```

**早期创业公司应该避免的 Claude Code 误用：**

```
❌ 让 Claude 帮你决定技术方向
   → 技术选型必须基于你们的具体约束，Claude 不了解

❌ 用 Claude 生成核心业务逻辑后不 review
   → 早期代码质量影响后续速度

❌ 过度依赖 Claude 补全测试来提高覆盖率
   → 测试要覆盖真实的业务风险，不是数字

✓ 用 Claude 快速探索新技术领域
✓ 用 Claude 生成重复性代码（CRUD、序列化）
✓ 用 Claude 做快速的代码 review
✓ 用 Claude 写内部文档和 runbook
```

**成本控制：**

早期公司 API 成本敏感。最有效的策略：
- 用 Claude.ai 订阅 + 手动粘贴（适合探索性工作）
- 高频重复任务用 API + 缓存
- 长上下文任务提前做 `/compact`

Claude Code 对早期创业公司的核心价值不是"生产力提升 X%"，而是让小团队能够**攻击更大的问题**——这才是创业公司真正需要的。
