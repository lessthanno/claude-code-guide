支付系统的开发比普通业务系统难，不是因为技术复杂，而是因为**错误的代价不对称**——少扣一笔钱是损失，多扣一笔钱是事故，丢失一笔是灾难。Claude Code 帮你在设计阶段就建立正确的心智模型。

**幂等性设计：**

幂等性是支付系统的基础——网络超时后重试，不能扣两次。

```bash
claude "帮我设计支付接口的幂等性方案：

场景：POST /api/payments 创建支付

问题：
- 网络超时，不知道服务器是否收到
- 用户双击提交按钮
- 第三方支付回调重复发送

方案要求：
1. 客户端传入 idempotency_key（UUID）
2. 服务器端如何存储和检查（Redis vs 数据库）
3. 相同 key 的重复请求返回相同结果
4. key 的过期时间如何设置
5. 如何处理'第一次还在处理中，第二次又来了'的竞态条件

给出 Go 实现（包括中间件和数据库 schema）。"
```

**状态机设计：**

```bash
claude "设计支付订单的状态机：

支付流程：
创建 → 待支付 → 支付中 → 已支付 → 已结算
              ↓
           已取消 / 支付失败 / 退款中 → 已退款

要求：
1. 哪些状态转换是合法的（有向图）？
2. 每个转换需要记录什么（谁触发、时间、原因）？
3. 如何防止并发导致的状态不一致（乐观锁 vs 悲观锁）？
4. 状态变更时如何触发下游（通知、对账）？

给出 PostgreSQL schema + Go 状态机实现。"
```

**对账系统设计：**

```bash
claude "设计我们的支付对账系统：

背景：
- 我们的系统：PostgreSQL 记录每笔交易
- 第三方支付：每天提供一个 CSV 对账文件
- 数据量：每天约 5 万笔

对账流程：
1. 下载第三方对账文件
2. 与本地记录比对
3. 找出差异：本方有对方无、对方有本方无、金额不一致
4. 差异分类：正常（时差）、异常（需要人工处理）
5. 生成对账报告

设计：
- 对账任务的 schema
- 差异分类的规则（什么算时差，什么算真正的不一致？）
- 异常的处理流程（自动修复 vs 人工审核）"
```

**支付安全的 checklist：**

```bash
claude "帮我完成支付模块的安全审查：

代码：[支付接口代码]

检查项：
1. 金额是否用整数（分）而不是浮点数？
2. 是否有 TOCTOU（检查后使用）漏洞？
3. 重放攻击防护（签名 + 时间戳）？
4. 敏感数据日志（卡号、CVV 不能出现在日志里）？
5. 金额篡改（前端传的金额 vs 后端实际应收）？
6. 并发超卖（库存检查和扣减是原子的吗）？
7. 异步通知的验签（第三方回调如何验真实性）？"
```

**退款和差错处理：**

```bash
claude "设计退款流程，考虑这些边界情况：

1. 已结算的订单如何退款（资金已到账）？
2. 部分退款（一个订单多件商品，只退一件）？
3. 退款超过原订单金额的防护？
4. 第三方退款接口超时，不知道是否成功？
5. 退款失败时的重试策略（指数退避）？
6. 退款成功后，用户账户、商户账户余额如何处理？

给出状态机图 + 关键流程的代码。"
```

**合规和风控的基础：**

```bash
claude "我的支付系统需要做基础的风控，帮我设计规则引擎：

基础规则：
- 单笔金额超过 5 万元，触发人工审核
- 同一用户 1 分钟内超过 3 笔，疑似异常
- 同一 IP 每小时超过 10 笔不同账号，疑似刷单
- 新注册用户（< 24 小时）大额支付，需要额外验证

实现要求：
1. 规则可配置（不需要改代码，改数据库就生效）
2. 规则计算要快（每笔支付的延迟 < 50ms）
3. 规则触发要有详细记录（方便事后分析）
4. 支持规则的灰度（新规则先对 10% 用户生效）"
```

支付系统的原则：**宁可拒绝合法交易，也不放过可疑交易**。这在业务上有代价，但在合规和资损上是正确的权衡。Claude Code 帮你把这些边界条件在开发阶段就想到，而不是在事故之后。
