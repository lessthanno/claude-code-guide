前端测试有两个极端：要么过度依赖手动测试（"我看着没问题"），要么写大量测试但都只测实现细节（改了变量名，100 个测试挂掉）。Claude Code 帮你找到中间地带：测试行为，不测实现。

**测试金字塔的前端版本：**

```bash
claude "帮我设计这个 React 应用的测试策略：

应用：任务管理 SPA
主要功能：创建/编辑/删除任务、拖拽排序、过滤/搜索

测试金字塔：
1. 单元测试（最多）：哪些适合单元测试？
2. 组件测试（中等）：哪些组件应该有独立测试？
3. E2E 测试（最少）：哪些关键路径必须 E2E 覆盖？

给出每层的典型测试用例，说明选择的理由。"
```

**组件行为测试（Testing Library 方式）：**

```bash
claude "帮我写这个搜索框组件的 Vitest 测试：

组件行为：
- 输入时，300ms 防抖后调用 onSearch
- 清空按钮在有内容时才出现
- 按 Escape 键清空内容
- loading 状态时禁用输入

测试原则：
- 用户行为驱动（用 userEvent 而不是 fireEvent）
- 通过 accessible name 查找元素，不用 className 或 id
- 不测内部实现（不测 debounce 的内部 timer）

给出完整测试文件，用 @testing-library/react + vitest。"
```

**自定义 Hook 的测试：**

```bash
claude "帮我测试这个 useLocalStorage hook：

interface useLocalStorage<T> {
  (key: string, initialValue: T): [T, (value: T) => void, () => void]
}
// 返回：[当前值, 设置函数, 删除函数]

测试覆盖：
1. 初始值（localStorage 为空时使用 initialValue）
2. 从 localStorage 读取已存值

```


前端测试的核心原则：**测试要能告诉你用户的体验是否正确，而不是代码的实现是否正确**。一个不需要改的测试，是对产品行为的文档；一个经常需要改的测试，通常是在测实现细节。Claude Code 帮你从"测什么"的问题开始，而不是"怎么写"。
