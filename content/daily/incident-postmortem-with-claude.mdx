大多数团队的事故复盘要么流于形式（写了但没人读），要么过于追责（找人背锅而不是找系统问题）。Claude Code 帮你快速生成高质量的 postmortem，重点放在系统性改进而不是个人失误。

**快速事故时间线重建：**

```bash
claude "帮我从这些日志重建事故时间线：

Slack 消息（已按时间排序）：
[粘贴 oncall 频道的消息记录]

相关日志片段：
[粘贴关键日志]

重建：
1. 事故开始时间（用户首次受影响）
2. 发现时间（谁/什么发现的）
3. 响应时间（第一个人介入）
4. 关键操作节点（做了什么，时间）
5. 恢复时间（服务恢复正常）

用标准格式输出，时间精确到分钟。"
```

**根因分析（5 Whys）：**

```bash
claude "帮我做这个事故的根因分析：

事故摘要：
- 2026-05-15 02:30，支付服务宕机 47 分钟
- 根本现象：数据库连接池耗尽
- 直接原因：某个新部署的批处理任务没有连接池限制

用 5 Whys 方法深挖：

Why 1：为什么数据库连接池耗尽？
→ 因为某个进程持有了大量连接不释放

Why 2：为什么这个进程持有大量连接？
→ ...

继续追问到根本的系统性原因（不是'因为有人写了 bug'）。
最终根因应该是可以通过流程或工具改进来预防的系统性问题。"
```

**Postmortem 文档生成：**

```bash
claude "基于以下信息生成规范的 postmortem 文档：

事故信息：
- 时间：2026-05-15 02:30 - 03:17 UTC+8
- 影响：支付服务完全不可用，47 分钟，影响约 2,000 笔待支付订单
- 触发原因：v2.3.4 版本上线后，批处理任务没有连接池限制
- 根因：代码审查流程没有检查数据库连接使用

格式要求（严格按照这个结构）：
1. Summary（一句话，不超过 50 字）
2. Impact（具体数字：时长、用户数、业务影响）
3. Timeline（按时间排序的关键事件）
4. Root Cause（系统性根因，不点名）
5. Contributing Factors（多个因素）
6. What Went Well（做对了什么，避免只写负面）
7. Action Items（具体可执行，有 owner 和 deadline）"
```

**Action Items 优先级评估：**

```bash
claude "这个 postmortem 有 12 条 action items，帮我优先级排序：

[列出 12 条 action items]

评估维度：
1. 防止同类事故再发（最高优先）
2. 减少 MTTR（Mean Time To Recovery）
3. 改善检测速度（从故障发生到发现的时间）
4. 降低实施成本

对每条：
- 优先级（P1/P2/P3）
- 理由
- 建议的 owner（角色而不是具体人）
- 建议的完成时间窗口（1 周/1 月/季度）"
```

**建立故障演练（Game Day）：**

```bash
claude "帮我设计一个 Game Day 演练方案，模拟最近这次数据库连接池耗尽的故障：

目标：
1. 验证 oncall 能在 10 分钟内发现并定位问题
2. 验证新的告警规则是否有效
3. 验证回滚流程是否清晰

演练设计：
- 如何安全地模拟连接池耗尽（不影响真实用户）
- 演练期间的观察者需要记录什么？
- 如何判断演练是否成功？
- 演练结束后的 debrief 怎么开？"
```

事故复盘的核心价值：**不是找到谁的错，而是找到系统的错**。好的 postmortem 的标准是：读完之后，任何一个合理的工程师都不会对涉及的人产生批评，因为在相同的系统和条件下，任何人都可能做出同样的决策。Claude Code 帮你把注意力保持在系统改进上。
