大多数人用 Claude Code 来生成代码，但它在**理解代码**上的价值往往被低估。Review 别人的代码、理解陌生系统、深挖某个技术决策——这些场景 Claude Code 同样极其有价值。

**理解陌生代码库：**

```bash
# 克隆一个感兴趣的开源项目
claude "帮我理解这个 Go HTTP 框架的核心机制：

我有一定 Go 基础，但从没用过这个框架。
代码：[粘贴核心文件]

不要解释每一行，帮我理解：
1. 框架解决了什么问题（和标准库相比）
2. 最关键的设计决策是什么，为什么这样设计？
3. 中间件系统的实现方式（与 Express.js 的对比）
4. 看完这个文件，我应该接下来读哪里？"
```

**深挖技术决策：**

```bash
claude "我看到这段代码用 sync.Pool 存储了 request 对象。

代码：[片段]

帮我理解这个决策：
1. sync.Pool 在这里解决什么问题？
2. 不用 Pool 的话，性能差多少？（有没有 benchmark 可以参考？）
3. 使用 Pool 时需要注意什么？（这段代码有遗漏吗？）
4. 什么时候不应该用 Pool？

我想把这个知识系统化，而不是只看一个例子。"
```

**Review PR 学习写法：**

```bash
# 看开源项目的高质量 PR
claude "帮我从这个 PR 中提取可以学习的工程实践：

PR 内容：[粘贴 diff 和评论]

我想学的不是'这个 PR 做了什么'，而是：
1. 作者的思维过程（为什么这样改？）
2. reviewer 的反馈揭示了什么经验（哪些是我想不到的）
3. 这个改动涉及的设计模式或原则
4. 如果我来实现，我可能会踩的坑"
```

**比较不同实现方式：**

```bash
claude "同样是实现 LRU 缓存，比较这两种 Go 实现：

实现 A（标准库）：
[代码]

实现 B（第三方库）：
[代码]

从这几个维度比较：
1. 算法复杂度（时间和空间）
2. 并发安全性（加锁粒度）
3. 代码可读性
4. 可扩展性（如果要加 LFU 怎么改）

哪种方式更适合生产使用？为什么？"
```

**从 bug 中学习：**

```bash
claude "这是一个生产 bug 的修复 commit：
[粘贴 commit diff 和信息]

帮我从中提取系统性的知识：
1. 这个 bug 的根本原因是什么？
2. 为什么测试没有发现它？（测试的盲区是什么？）
3. 修复方案的 tradeoff（有没有其他更好的修复方式？）
4. 这类 bug 有什么通用的预防模式？
5. 我应该在我的代码库里检查哪些类似的地方？"
```

**建立自己的代码知识库：**

```bash
claude "帮我整理这个月 code review 中遇到的问题，形成可复用的 checklist：

我遇到的问题列表：
1. 并发 map 操作没有加锁
2. error 直接 fmt.Sprintf 丢失了 error chain
3. context.Background() 用在了应该传 ctx 的地方
4. defer 在循环里导致资源没有及时释放
...

帮我：
1. 把这些分类（并发、错误处理、资源管理等）
2. 找出每类问题背后的同一个根因
3. 总结成 10 条 review checklist（可以加到 CLAUDE.md）
4. 哪些问题可以用 linter 自动检测？"
```

Code review 作为学习的黄金法则：**不要只看代码做了什么，要看它为什么这样做**。一个改动背后的上下文、考量、和 tradeoff，比改动本身更有学习价值。Claude Code 帮你把代码阅读从"看懂了"变成"理解了"。
