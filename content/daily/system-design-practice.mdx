系统设计能力不是背答案，而是建立一套**从需求到架构决策的推理框架**。Claude Code 是最好的练习伙伴——它既能配合你演练，也能质疑你的假设。

**结构化的系统设计对话：**

```bash
claude "我们来做系统设计练习：设计一个短链接服务（类 bit.ly）。

我扮演候选人，你扮演严格的面试官：
1. 当我遗漏重要的需求澄清时，提醒我
2. 当我的估算有明显错误时，纠正我
3. 当我跳过重要的设计决策时，追问
4. 每隔 5 分钟提示我时间（总共 45 分钟）

从提问需求开始，我先问你问题。"
```

**需求澄清训练：**

```bash
claude "练习需求澄清。我要设计'消息通知系统'。

帮我列出在设计之前必须澄清的问题（10 个最重要的），
以及每个问题的答案如何影响设计决策：

例如：
Q：通知需要保证送达吗（at-least-once vs at-most-once）？
A 选项1（保证送达）：需要持久化队列、ack 机制、重试
A 选项2（尽力送达）：可以用 pub/sub，简单很多

给出 10 个类似这样的问题和设计影响。"
```

**容量估算实战：**

```bash
claude "帮我做 Twitter 级别的 Feed 系统容量估算：

已知：
- DAU：1 亿用户
- 每用户每天发 1 条推文
- 每用户每天读 100 条推文
- 每条推文平均 140 字节

推算：
1. 写入 QPS（推文写入）
2. 读取 QPS（Feed 读取）
3. 存储量（每天、每年）
4. 带宽（读写各自）

每步给出计算过程，最后说明这些数字对架构选型的影响。"
```

**架构方案对比：**

```bash
claude "Feed 系统的核心设计决策：Pull vs Push（Fan-out on read vs Fan-out on write）

对每种方案分析：
1. 写路径的实现
2. 读路径的实现
3. 存储需求（差异）
4. 延迟特性（写延迟 vs 读延迟）
5. 适用场景（何时选哪种）

特殊情况：
- 大 V 用户（百万关注者）如何处理？
- 用户不活跃时，pre-computed feed 浪费资源怎么办？

推荐哪种方案？为什么？"
```

**深挖设计细节：**

```bash
claude "短链接服务中，如何生成唯一的短码？

比较这些方案：
1. UUID 截断（取前 6 位）
2. 随机字符串 + 碰撞检测
3. 全局自增 ID + Base62 编码
4. 哈希（MD5/SHA1 截断）

对每种：
- 唯一性保证（碰撞概率）
- 生成速度
- 分布式环境下的问题
- 是否有信息泄露风险（能推断出 ID 序列）

最终推荐，以及如何保证分布式下的 ID 唯一性。"
```

**故障模式分析（FMEA）：**

```bash
claude "对这个设计做故障模式分析：

设计：短链接服务
- Nginx → API 服务（3 台）→ Redis（缓存短链接）→ MySQL（持久化）

分析每个组件失败时的系统行为：
1. Redis 单节点故障？
2. MySQL 主库故障？（主从切换时间？期间的请求怎么处理？）
3. 一台 API 服务器崩溃？
4. Nginx 崩溃？
5. 网络分区（Redis 和 MySQL 无法通信）？

对每种故障：
- 用户看到的现象
- 系统自动恢复还是需要人工介入
- 数据是否会丢失或不一致"
```

系统设计能力提升的最快路径：**主动找到设计决策中的 tradeoff，不是找'正确答案'**。没有普遍正确的架构，只有在给定约束下更好的选择。Claude Code 能帮你深挖每个选择的代价，这比背架构图有价值得多。
